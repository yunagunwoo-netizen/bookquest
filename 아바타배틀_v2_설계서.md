# 아바타 토론배틀 v2 — 구현 설계서

**목표:** 가위바위보식 RPS 배틀(Phase 5b MVP)을 **아바타끼리 자동 진행되는 지식·토론 배틀**로 교체. 아이는 채팅창 형태로 관람만. 주1회 무료.

**작성일:** 2026-04-22
**엔진:** Gemini 2.5 Flash (AI 코치에서 검증된 저비용 모델 재활용)

---

## 1. 핵심 컨셉 (user decisions 반영)

| 항목 | 결정 |
|------|------|
| 배틀 포맷 | 하이브리드 — Q&A 2라운드 + 미니토론 1라운드 |
| 플레이어 입력 | **없음** (아바타 자동 진행) |
| 관람 방식 | 채팅창 스타일 메시지 스트리밍 |
| 질문 대상 (Q&A) | 해당 시즌 12권 전체 |
| 질문 대상 (토론) | 책 내용 한정 X — 아바타가 쌓은 지식 기반 |
| 매칭 | 비슷한 레벨 랜덤 |
| 빈도 | 주1회 무료 (일1회는 유료 전용으로 추후) |
| 결과 | 승리 시 랭킹 가점·코인, 채팅 로그 저장(리플레이) |

**설계 철학:** 매 배틀이 곧 리플레이. 아이는 관찰자. "이 배틀을 보는 것만으로도 공부가 된다"는 passive learning 구조.

## 2. 백엔드 — Cloud Function `runAvatarBattle`

### 2.1 시그니처
```
POST /runAvatarBattle
Body: { myUid, oppUid }
Auth: Firebase Auth 필수
Response: { battleId, winner: "me"|"opp"|"draw", rounds: [...] }
```

### 2.2 1회 LLM 호출로 전체 배틀 시뮬레이션

비용 최적화를 위해 **단일 Gemini 호출**로 3라운드 전체 생성:

```
System prompt:
  너는 아이들 독서앱 "BookQuest"의 아바타 배틀 중계 엔진이다.
  두 아바타의 지식 수준·독서 이력을 바탕으로 3라운드 배틀을 시뮬레이션해.
  결과는 다양하게 나와야 재미있다 — 레벨 차이가 작으면 역전도 가능.

User prompt (JSON):
  {
    "myAvatar": {
      "name": "건우의 루미", "level": 15, "streak": 28일,
      "completedBooks": ["책제목1", "책제목2", ...],
      "recentQuizWins": 12, "discussionStars": 8
    },
    "oppAvatar": { ...같은 구조 },
    "seasonBooks": ["시즌12권 제목 리스트"],
    "format": "q2_debate1"
  }

Expected output (JSON):
  {
    "rounds": [
      { "type": "qa", "book": "...", "question": "...",
        "myAnswer": "...", "oppAnswer": "...",
        "judge": "my|opp|draw", "reason": "..." },
      { "type": "qa", ... },
      { "type": "debate", "topic": "...",
        "myStance": "...", "oppStance": "...",
        "myArgument": "...", "oppArgument": "...",
        "judge": "my|opp|draw", "reason": "..." }
    ],
    "winner": "my|opp|draw",
    "narration": "전투 전체 내레이션 1~2줄"
  }
```

**예상 토큰:** 입력 1,500 + 출력 2,500 = 배틀당 **약 5원** (Gemini 2.5 Flash, 환율 1,350)
→ 엑셀 모델의 9원보다 낮음 (단일 호출 효율)

### 2.3 저장 (Firestore)

```
battles/{battleId}:
  myUid: string
  oppUid: string
  myAvatarSnapshot: { name, level, avatar, streak, titleId }
  oppAvatarSnapshot: { ... }
  rounds: [...3 round objects]
  winner: "my"|"opp"|"draw"
  narration: string
  createdAt: Timestamp
  seasonId: string  // 나중에 시즌별 통계용
```

### 2.4 유저 문서 업데이트
```
users/{myUid}:
  battleWins / battleLosses / battleDraws ++
  weeklyBattleWins ++ (승리 시)
  lastBattleWeek: "2026-17"  // ISO week 문자열
  lastBattleAt: Timestamp
```

## 3. 프론트엔드 — index.html 수정

### 3.1 제거
- `BATTLE_SKILLS` 상수 (RPS 스킬 3종)
- `pickOpponentSkill()` 함수
- `judgeBattle()` 함수
- `battleMySkill`, `battleOppSkill` 상태
- BattleScreen 내 "스킬 선택" UI 블록 (ready/reveal 단계)

### 3.2 유지 (그대로 사용)
- `battleWins`, `battleLosses`, `battleDraws`, `weeklyBattleWins` 상태
- 주간 리셋 로직 (월요일 기준 weeklyBattleWins=0)
- `fetchRanking()` 의 score 공식 `weekly * 1.0 + weeklyWins * 0.3`
- 매칭 로직 (비슷한 레벨 랜덤)

### 3.3 신규 추가
- `battleReplay` 상태: 서버에서 받은 rounds 배열
- `battleStreamIndex` 상태: 현재 스트리밍 중인 메시지 인덱스
- `canBattle()` 헬퍼: 현재 주에 이미 배틀했는지 체크
- 채팅창 스타일 메시지 뷰 (카카오톡 말풍선 느낌)
- 메시지 한 줄씩 딜레이로 등장 (typing 효과)
- 라운드 사이 구분 배너 ("라운드 2", "마지막 라운드: 토론!")
- 전체 재생 완료 후 결과 카드

### 3.4 새 BattleScreen 단계 (`battleStep`)
```
"gate"      — 이번 주 이미 배틀함, 남은 시간 안내
"matching"  — 매칭 중 스피너
"ready"     — 상대 만남, "시작" 버튼 (또는 자동 시작)
"playing"   — 채팅창 스트리밍
"result"    — 결과 카드 + 공유 버튼
```

### 3.5 리플레이 (Phase 2 — 이번 구현엔 포함 X)
- 내 지난 배틀 목록 (최근 5전)
- 랭킹 상위 유저의 공개 배틀 관람

## 4. Firestore Security Rules

```
match /battles/{battleId} {
  allow read: if request.auth != null
    && (resource.data.myUid == request.auth.uid
        || resource.data.oppUid == request.auth.uid);
  allow write: if false;  // 서버 전용 (Cloud Function으로만 생성)
}
```

## 5. 주1회 제한 구현

```javascript
// 클라이언트 검증 (서버 검증도 필수)
function canBattle() {
  const now = new Date();
  const week = getIsoWeek(now);  // "2026-17"
  return lastBattleWeek !== week;
}

// Cloud Function 서버 검증
if (userDoc.data().lastBattleWeek === currentWeek) {
  throw new HttpsError("resource-exhausted",
    "이번 주 배틀을 이미 사용했어요. 다음 주 월요일에 돌아올게요!");
}
```
