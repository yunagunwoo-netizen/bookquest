## F. 주의사항 (오늘 겪은 버그와 해결법)

### F-1. ⚠️ `index.html` tail truncation 반복 발생 (오늘 3번)
**현상**: Python의 `f.write(large_string)` 호출 후 파일 끝 ~1~3KB가 사라지고 `</html>` 누락.

**원인 추정**: Windows 마운트 경유 대용량 파일(1.7MB+) 쓰기에서 버퍼 flush 전에 반환되는 파일시스템 레이어 버그.

**해결 패턴**:
```python
import os
with open(PATH, 'w', encoding='utf-8') as f:
    f.write(src)
    f.flush()
    os.fsync(f.fileno())  # 🔑 핵심: 물리 디스크까지 강제 flush

# 쓰기 후 즉시 재검증
with open(PATH, 'r', encoding='utf-8') as f:
    check = f.read()
assert check.rstrip().endswith('</html>'), 'tail truncation!'
```

**재발 시 복구 절차**:
```python
with open('index.html', 'r') as f: cur = f.read()
idx = cur.rfind('setInstallPrompt(null);')
cur = cur[:idx + len('setInstallPrompt(null);')] + """<install banner 및 SW 등록 블록 복구 템플릿>"""
# fsync로 저장
```
설치 배너 + SW registration + `</body></html>` 템플릿은 이 문서의 과거 백업 파일 또는 `index.html.pre_domain_icons_backup` 에서 복구 가능.

**`deploy_bookquest.ps1` Safety guard가 이미 이 체크 포함**이라 깨진 파일이 git push 되지는 않음. 하지만 로컬 파일은 손상 상태로 남을 수 있음.

### F-2. ⚠️ Edit 툴을 `index.html`에 쓰지 말 것
- 4.76MB 파일일 때도 truncation 이력. **Python byte-level replace만 사용** (반드시 fsync + 재검증)
- 규모가 크면 반드시:
  1. replacement anchor count 검증 (`n = src.count(old); if n != 1: abort`)
  2. 쓰기 후 `fsync` + 재읽기
  3. `</html>` 종료 확인
  4. Babel 파싱 (node + `@babel/parser`)
